## COMMON PITFALLS TO AVOID                                                     
                                                                                
1. **Bitstream mismatch**: If you change .v/.luc files, you MUST re-synthesize. 
   Uploading new firmware against an old bitstream will crash unpredictably.    
                                                                                
2. **UART TX "hang"**: simpleuart stalls the CPU via mem_ready until TX         
completes.                                                                      
   This is NORMAL BEHAVIOR, not a bug. Each character takes ~10us at 921600     
baud.                                                                           
   printf of a long string will be slow but should not hang indefinitely.       
                                                                                
3. **Stack overflow**: PicoRV32 stack starts at 0x00001000 (top of 4KB BRAM)    
and                                                                             
   grows DOWN. DOOM uses significant stack space. If the stack underflows into  
   address 0x00000000, it overwrites the reset vector and the CPU crashes.      
   This is a possible cause of the 0xFF crash.                                  
                                                                                
4. **Heap configuration**: _sbrk must return addresses in the DDR3 heap region  
   (0x20600000-0x20DFFFFF). If it returns addresses in BRAM or outside DDR3,    
   DOOM will corrupt memory and crash.                                          
                                                                                
5. **WAD file access**: fpga_file.c maps doom1.wad to 0x20100000. If the WAD    
   wasn't uploaded correctly (wrong offset, truncated transfer), DOOM will      
crash                                                                           
   during W_Init when it tries to read WAD headers.
